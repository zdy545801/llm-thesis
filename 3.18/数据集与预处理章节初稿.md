# 数据集与预处理章节初稿

## 1. 数据来源

### 1.1 CrowS-Pairs
- 关键字段：`sent_more`、`sent_less`
- 任务目标：比较模型对刻板句与中性句的偏好差异

### 1.2 BBQ
- 关键字段：`context`、`question`、`ans0/ans1/ans2`、`label`
- 任务目标：观察编辑前后多选问答中的偏见相关变化

### 1.3 编辑数据
- 文件：`data/edits_toy.json`
- 作用：模拟连续编辑请求，构造轮次化实验

## 2. 本文使用的数据路径
- `data/crows_pairs_sample.jsonl`
- `data/bbq_sample.jsonl`
- `data/edits_toy.json`

说明：当前阶段使用 sample 子集，优先保证流程跑通与可复现；GPU恢复后可扩展样本规模。

## 3. 预处理策略
1. 统一 UTF-8 读取。  
2. CrowS：过滤缺失句对样本。  
3. BBQ：过滤缺失标签或问题样本。  
4. 编辑集：若无 `subject`，基于 `src` 规则提取主体（用于编辑接口）。

## 4. 评估输入构造

### 4.1 CrowS 评估输入
对同一条样本的两句分别计算困惑度：
- `PPL(sent_more)`
- `PPL(sent_less)`

据此统计 `prefer_stereo_rate`。

### 4.2 BBQ 评估输入
对每个选项构造文本：
`Context + Question + Answer: option`

比较三个选项困惑度，最低者作为预测，得到 `accuracy_proxy`。

## 5. 一致性控制
- 全部轮次使用同一模型版本和同一 tokenization。  
- 固定轮次命名：`round_0.json`, `round_1.json`, ...  
- 输出统一目录：`results/rounds/`，便于后处理脚本批量计算。

## 6. 局限说明
- sample 规模较小，统计显著性有限。  
- BBQ 当前为代理式评估，不替代官方完整流程。  
- 现阶段重点是“趋势可观察、流程可复现”。
